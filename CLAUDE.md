## Commands

```bash
pnpm dev              # Start Vite dev server (localhost:3000)
pnpm build            # Production build
pnpm start            # Run production server (after build)
pnpm test             # Run all tests (vitest run)
pnpm lint             # ESLint
pnpm typecheck        # tsc --noEmit (not in package.json; use: npx tsc --noEmit)

# Run a single test file
pnpm test src/stores/chat-store.test.ts

# Electron
pnpm electron:dev     # Dev mode with Electron shell
pnpm electron:build   # Package Electron app

# Cloudflare Worker (playground)
cd playground-ws-worker && pnpm dev
```

**Node ≥22 and pnpm ≥10.24 required.** Use `pnpm` — not npm/yarn.

## Architecture Overview

Hermes Workspace is a React 19 + TanStack Start (SSR) frontend that acts as a control panel for **Hermes Agent**, an external AI gateway process. The workspace connects to the gateway via HTTP + WebSocket at `HERMES_API_URL` (default: `http://127.0.0.1:8642`).

### Request flow

```
Browser → TanStack Router (file-based, src/routes/) 
       → API routes (src/routes/api/**) 
       → src/server/* handlers 
       → Hermes Agent gateway (WebSocket/HTTP)
```

### Key layers

**Routing & SSR** — `src/router.tsx` bootstraps TanStack Router; `src/routes/__root.tsx` is the root layout. Routes are file-based; `routeTree.gen.ts` is auto-generated by Vite. SSR is enabled for chat, sessions, jobs, and tasks routes via the TanStack Start Vite plugin.

**Server logic** (`src/server/`) — Node.js-side code only. Critical files:
- `gateway.ts` — WebSocket/HTTP client to Hermes Agent with crypto auth
- `auth-middleware.ts` — Password/token auth; every route is protected
- `dashboard-aggregator.ts` — Real-time metrics (sessions, cost, model mix)
- `hermes-config-store.ts` — Persistent workspace configuration
- `mcp-tools-cache.ts` / `mcp-presets-store.ts` — MCP catalog management

**State management** (`src/stores/`) — Zustand. One store per concern: `chat-store`, `task-store`, `mission-store`, `workspace-store`, `agent-swarm-store`, `session-model-store`.

**Screens vs Components** — `src/screens/` holds full page-level views (agents, chat, crew/swarm, dashboard, playground, etc.). `src/components/` holds reusable pieces. This distinction matters: screens are route-mounted, components are composed inside screens.

**MCP integration** — The `mcp-input.ts` types file is server-only. An ESLint rule (`eslint.config.js`) explicitly forbids importing it in client code.

### Key environment variables

| Variable | Default | Purpose |
|---|---|---|
| `HERMES_API_URL` | `http://127.0.0.1:8642` | Gateway backend |
| `HERMES_API_TOKEN` | — | Gateway auth token |
| `HERMES_PASSWORD` | — | UI password (required for non-loopback) |
| `HOST` | `127.0.0.1` | Bind address |
| `PORT` | `3000` | Server port |
| `HERMES_AGENT_PATH` | — | Path to hermes-agent binary |

Copy `.env.example` to `.env` to configure locally.

### Deployment targets

- **Docker** — `docker/Dockerfile`, runs as non-root `workspace` user, requires Python 3 for PTY support
- **Electron** — `electron/` directory; packaged via `electron-builder.config.cjs` for macOS (arm64/x64) and Windows
- **Cloudflare Worker** — `playground-ws-worker/` with `wrangler.jsonc`

### Path alias

`@/*` maps to `./src/*` (configured in both `tsconfig.json` and `vite.config.ts`).
